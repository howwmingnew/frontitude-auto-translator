**API Dispatching (lines 2906-2912):**
```javascript
function callTranslateAPI(texts, targetLang) {
  switch (state.provider) {
    case 'openai': return callOpenAI(texts, targetLang);
    case 'gemini': return callGemini(texts, targetLang);
    default: return callDeepL(texts, targetLang);
  }
}
```

**DeepL Integration (lines 2981-3010):**
- `callDeepL(texts, targetLang)`: POST to `https://api.deepl.com/v1/translate` or `:fx` (free)
- Detects free key by `:fx` suffix, routes to correct endpoint
- Converts language codes (en → EN, zh → ZH-HANS, etc.)

**OpenAI Integration (lines 3014-3058):**
- `callOpenAI(texts, targetLang)`: POST to `https://api.openai.com/v1/chat/completions`
- JSON array prompt: "Translate these texts to [language]: [texts]" → expect JSON array back
- Uses selected model (gpt-5.4-nano, gpt-5.3-turbo, gpt-5-mini, etc.)
- Includes context prompt in system message if provided
- Handles temperature variations (not supported on some models)

**Gemini Integration (lines 3060-3091):**
- `callGemini(texts, targetLang)`: POST to `https://generativelanguage.googleapis.com/v1beta/models/...`
- Same JSON array prompt structure as OpenAI
- Strips markdown code fences from response

**Download/Export (lines 3111-3126):**

## Naming Conventions

**Files:**
- Single file: `index.html` (not `app.html`, `translator.html`)
- Documentation: `README.{language}.md`
- Configuration: `.claude/settings.local.json`, `CLAUDE.md`

**Functions:**
- camelCase: `handleFileSelect()`, `renderEditorTable()`, `callOpenAI()`
- Verbs first: `get*`, `set*`, `render*`, `handle*`, `call*`, `update*`
- Private (closure scope): all functions, no explicit `_` prefix

**Variables:**
- camelCase: `state`, `dom`, `provider`, `jsonData`
- CONST_CAPS for immutable config: `UI_TRANSLATIONS`, `PROVIDER_CONFIG`, `LANG_CODE_TO_NAME`, `MAX_FILE_BYTES`, `FETCH_TIMEOUT_MS`
- Collections: plural names (`languages`, `textsToTranslate`, `editedCells`)
- Booleans: prefix with `is`, `has`, `should`: `isEmpty`, `isFullyDone`, `hasError`

**HTML Elements:**
- kebab-case IDs: `#upload-card`, `#provider-select`, `#editor-table`
- Data attributes: `data-i18n`, `data-i18n-html`, `data-lang`, `data-theme`
- CSS classes: kebab-case with BEM-like prefixes: `sidebar-drawer`, `card-title`, `btn btn-primary`

**State Keys:**
- Flat structure (no nested objects except jsonData, editedCells Map)
- Examples: `uiLang`, `translating`, `contextPrompt`, `fullyTranslatedLangs`

**CSS Variables:**
- Semantic naming: `--primary`, `--error`, `--text`, `--bg`
- Modifiers: `--primary-hover`, `--primary-light`, `--primary-muted`, `--primary-border`
- Shadows: `--shadow-sm`, `--shadow-md`, `--shadow-lg`

## Where to Add New Code

**New Feature (e.g., "Add French translation support"):**
- UI Translations: Add keys to `UI_TRANSLATIONS` object (all languages)
- Update `langCodeToName()` if new language display name needed
- Add provider model if API requires specific handling
- Implement: No code changes usually needed if using standard ISO 639-1 code

**New Translation Provider:**
- Add to `PROVIDER_CONFIG` object with model definitions and endpoints
- Implement `call{Provider}()` function following OpenAI/DeepL/Gemini pattern
- Update `callTranslateAPI()` switch statement
- Update UI dropdowns (auto-populated from PROVIDER_CONFIG)
- Add provider-specific language code mapping if needed

**New UI Component (e.g., "Add statistics panel"):**
- Add HTML markup in appropriate section (before `<script>`)
- Add CSS styles in `<style>` block
- Add DOM reference in `dom` object
- Add event listeners near other handlers of that type
- Update i18n keys in all languages
- Use immutable state updates (setState) for reactive changes

**New Modal or Overlay:**
- Add HTML markup after existing modals (lines ~1530-1562)
- Add corresponding CSS styles
- Implement open/close functions following cell-edit-modal pattern
- Add DOM references to `dom` object

**Bug Fix or Small Enhancement:**
- Identify affected layers: UI/State/Event/API/Utility
- Locate code section using function names or line numbers
- Follow existing patterns: camelCase names, immutable updates, try-catch error handling
- Test in browser (no build step needed)
